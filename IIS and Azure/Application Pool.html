All About Application Pool

Application pools allow you to isolate your applications from one another, even if they are running on the same server.

Application Pools use the Server Process called w3wp.exe.

Shared Application Pool

Shared Application Pool means you have hosted many web applications in the same application pool. Say  5 websites hosted on IIS, then all share the same Application Pool.

Benefits:

  • It saves server memory(RAM) and other resources

Disadvantages:

  • If there is an error in one app and takes more ram, it will obviously down other applications. So, if there is no significant reason of splitting the application then its better to keep in same application pool.

Dedicated Application Pool

This means that each web application has its own application pool so separate instances of W3wp.exe process.

Advantages:

  • Crashing one application will not affect other application performance because each will be running in its own application pool. So, better performance.

  • By isolating applications, you reduce the chance that one application will access the resources of another application.

  • You can run untrusted and third-party applications in a very restrictive environment so they do not get unauthorized access to other applications.

Disadvantages 

  • Consumes more ram and resources.

Look multiple application has separate Application Pool.

 

  • There will be many application pool in IIS.You can also create application pool.
  • Each pool can hold multiple applications.
  • Each application pool by default has 1 worker process. So, if many request comes then there are queue and are not rejected.
  • If you increase the count of worker process more than one then it is called Web Garden.
    For this you need to take care if there is in-proc session management.

Find Number of Worker Process Assigned to Application Pool

  1. Right click the application pool .
  2. Select Advanced Settings.
  3. Find /Increase the Maximum Worker Processes Count.

Static Variable Life Time is related to Application Pool

  • Static variable is application level .There lifetime exists as long as application closed.

     

  • Static fields are one instance per app-domain meaning all requests here the same value.

  • By Default, if the application itself doesnot receive any request for 20 minutes,it will unload from memory.All static variables will then be lost; on the next request to the
    app, a new app instance will get loaded.

    Static variables are lost when IIS restarts your asp.net application when any of the below happens:
    1) Manual restart of app pool
    2) The pool has reached some limits that is defined on the server and restart.
    3) Restart IIS
    4) If application remains idle for 20 minutes no request, then by default worker process working for the application also get closed.

    5) Reboot server.
